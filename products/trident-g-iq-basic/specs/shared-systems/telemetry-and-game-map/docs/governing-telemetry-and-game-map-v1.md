# **Governing Telemetry & Game Map v2**

## **A. Purpose**

This revision updates the Capacity Gym governing telemetry and game map so that it supports **true motor habit formation** rather than short-burst score chasing.

The central change is non-negotiable:

**The compulsory entry route is 20-day â†’ 10-day â†’ 3-day, not the reverse.**

The 20-day block is the encoding phase in which the user proceduralises the core executive control pattern under variable practice and speed stress. The 10-day block is a consolidation phase for deepening one lane once the portable core exists. The 3-day block is a maintenance, reactivation, or pre-load format that only becomes available after the 20-day foundation has been completed.

This keeps the product aligned with Trident-Gâ€™s portability-first logic:

**stabilise â†’ perturb â†’ select invariants â†’ bank â†’ deploy**

and prevents users from mistaking brief local performance gains for installed, wrapper-robust competence.

## **B. Canonical correction to the game map**

The Capacity Gym game set should be treated as containing **5 families and 13 variants**:

- **Family 1: XOR** â€” 3 variants

- **Family 2: AND** â€” 2 variants

- **Family 3: Interference** â€” 3 variants

- **Family 4: Emotional** â€” 2 variants

- **Family 5: Relational** â€” 3 variants

These all share the same n-back backbone but foreground different skill clusters:

- **XOR**: flexibility, selective attention, conceptual abstraction

- **AND**: feature binding, associational memory

- **Interference**: inhibitory control, conflict resolution, conceptual abstraction

- **Emotional**: affective working memory, emotional regulation, hot-state filtering

- **Relational**: relational integration, fluid reasoning

The progression engine remains shared. What changes is the foreground skill demand, wrapper library, and later proof profile.

## **C. Non-negotiable sequencing rule**

### **C1. Phase order**

The programme order is:

1.  **Phase 1 â€” Encode**

    - **20-Day Rewire**

    - mandatory first phase

    - builds the portable executive motor pattern

2.  **Phase 2 â€” Consolidate**

    - **10-Day Programme**

    - unlocks only after Phase 1 completion

    - deepens one family or branch while preserving the core

3.  **Phase 3 â€” Maintain / Reactivate**

    - **3-Day Sprint**

    - unlocks only after Phase 1 completion

    - used for reactivation, pre-load, or diagnostic maintenance

### **C2. Why the order matters**

Three days is too short for procedural consolidation. Used first, it is likely to produce brittle, wrapper-bound performance rather than automaticity that survives variation.

The 20-day block is therefore the **encoding window** in which the user learns the executable syntax of the core control pattern under repeated stabilisation, wrapper variation, and speed stress.

Only after that core is encoded should the app offer shorter specialist or maintenance formats.

## **D. Session structure**

Each daily session contains:

- **10 blocks**

- **20 + n trials per block**

- **2 speed settings**

  - **Normal** = 3.0s ISI

  - **Fast** = 1.5s ISI

The block structure remains unchanged. What changes is how progression and programme access are governed.

## **E. Governing metrics**

The MVP should still use two main progression metrics:

1.  **Last-3-block consistency**

2.  **Average session n-back**

These remain the cleanest core governing metrics.

### **E1. Last-3-block consistency**

This remains the primary **within-session routing** signal.

A last-3-block run counts as consistent when:

- at least 2 of the final 3 blocks are at the same n-level

- the third block is no more than 1 n-level away

- there is no obvious late-session collapse

This is a stability gate, not a peak-score gate.

### **E2. Average session n-back**

This remains the primary **session-level progress** signal.

It is used for:

- cross-session trend tracking

- plateau detection

- setting start level for guided probes after Phase 1

- broader summaries of progress through the programme

### **E3. Additional required gate: speed status**

Speed does **not** need to become a separate third headline metric, but it must become a **mandatory gate condition** for progression.

The reason is simple: if the user can only perform the task at comfortable pacing, the control pattern may still be controlled rather than procedural.

So the revised rule is:

- **variant swap remains the first portability pressure**

- **fast-speed hold becomes the next required pressure**

- **n-increase follows only after stability under those pressures**

This preserves the original coach order while making the speed gate behaviourally stricter.

## **F. Revised progression priority**

The coachâ€™s default priority should be:

1.  **switch variant**

2.  **hold the level at fast speed**

3.  **increase n**

This is now a stricter rule, not just a preference.

### **F1. Why this order is kept**

- **Variant swap** imposes variable practice and tests whether the skill survives wrapper change.

- **Fast speed** forces the policy towards automatic execution rather than slow conscious compensation.

- **Higher n** should come only after the lower-level control pattern has shown portability and speed robustness.

In other words, the system should reward:

**portable control first, automaticity second, raw difficulty third.**

### **F2. Credit weighting**

The current weighting can stay, but should now be interpreted more explicitly as a motor-habit architecture:

- **wrapper / variant swap** = 30

- **fast-speed hold** = 20

- **next n-level attempt** = 10

Credits are awarded only when supported by acceptable last-3-block consistency.

This keeps the app from rewarding harder-looking choices that are not yet stabilised.

## **G. Family mastery and unlock rule**

### **G1. Access states**

Each family still has three states:

- **Locked**

- **Probe unlocked**

- **Permanently unlocked**

### **G2. Permanent family unlock**

A family should no longer be treated as cleared by normal-speed 3-back alone.

A family is **permanently unlocked** only when the user shows:

1.  **stable 3-back across all variants in that family**, and

2.  **at least one successful fast-speed confirmation at that level inside the family**, and

3.  **at least one same-level hold after wrapper change inside that family**

This is a cleaner motor-habit interpretation of family mastery:

- **stable level** proves the policy exists

- **wrapper hold** suggests it is not tied to one surface form

- **fast-speed hold** suggests it is becoming procedural rather than merely deliberate

### **G3. What unlock never means**

- plateau does **not** grant permanent unlock

- curiosity does **not** grant permanent unlock

- one spike does **not** grant permanent unlock

Permanent unlock is a **validated portability gate**, not a novelty reward.

## **H. Revised developmental backbone**

The previous flexible time-frame-first structure should be replaced by a clearer developmental ladder.

### **H1. Core backbone during Phase 1**

The 20-day encoding block should use the fixed core route:

**XOR â†’ AND â†’ Interference**

These three families form the core executive staircase:

- **XOR** = remapping and selective feature control

- **AND** = active feature binding

- **Interference** = stronger inhibitory and conflict control

### **H2. Advanced branches after Phase 1**

The following should unlock only after the 20-day foundation is complete:

- **Emotional**

- **Relational**

These are not ideal entry families for motor habit formation.

- **Emotional** adds hot-state salience and affective interference pressure

- **Relational** pushes the system towards deeper abstraction and reasoning

They make better sense as **post-foundation extensions** once the core executive syntax has been proceduralised.

### **H3. What to remove**

Remove the earlier â€œelite challengeâ€ framing that encouraged trying to unlock all five families inside the initial 20 sessions.

That structure incentivises breadth before procedural depth and risks overload, brittle learning, and score-chasing.

## **I. Programme architecture**

## **I1. Phase 1 â€” 20-Day Rewire (mandatory)**

### **Purpose**

Encode the portable executive motor pattern through repeated stabilisation, wrapper variation, and speed stress.

### **Product role**

This is the **mandatory entry route** for any new user who has not yet built a stable Capacity Gym foundation.

### **Main training logic**

- start with the core family path

- progress via wrapper swap first

- require fast-speed confirmation before major advancement

- use plateau probes only as controlled perturbation, not as shortcut access

- treat stable late-session holds as the main evidence of usable consolidation

### **Primary goal**

Build a **portable core** that can survive:

- wrapper variation

- modest speed pressure

- controlled family progression

### **End-state target**

By the end of the 20-day Rewire, the user should ideally show:

- stable 3-back in the current cleared core family or families

- wrapper-robust carry-over inside those families

- at least some fast-speed confirmation

- reduced collapse under late-session load

- improving portability markers rather than only rising peaks

### **Recommended user-facing wording**

**20-Day Rewire: Build the core first**

### **Main Transfer Readiness interpretation**

This block is the first serious route to **Transfer Readiness = Broadening**, because it is the first phase in which the system can plausibly claim early procedural carry-over rather than mere exposure.

## **Critical Constraint for the 20-Day Phase**

During the mandatory 20-Day Encode phase, probes must follow these stricter rules:

1.  Temporary only: Probe access lasts max 2 sessions, then mandatory return to origin family (Section 11)

2.  Core backbone only: Even if probing, stick to XORâ†’ANDâ†’Interference path (Section K3: "probes should remain subordinate to the fixed backbone")

3.  No permanent unlocks: Do not permanently unlock the probe family during the 20-day block, even if they stabilize there. The 20 days is a temporal anchor for consolidation, not just a content checklist.

4.  Diagnostic routing: If the probe session ends with goodSession == true (last-3 consistent), allow one more probe session. If not, immediate return to origin.

## **M. Transfer Readiness wording**

Keep the user-facing broader carry-over metric as:

**Transfer Readiness**

not **Far Transfer**.

This remains the correct modest wording because the stronger Trident-G threshold for reliable far transfer requires more than in-app wrapper carry-over. It requires broader portability, cue-fired deployment, repeated validation, and conservative banking.

In Capacity Gym, Transfer Readiness should therefore be interpreted as:

- **Emerging**

  - first stable carry-over inside a family

- **Developing**

  - repeated carry-over with lower swap cost and less collapse

- **Broadening**

  - stable portable core across the phase backbone, including speed and wrapper robustness
