The core principle is simple: **only clean in-band full sessions count towards the 20-session encoding phase**. That follows the Trident rule that learning should be intensified only when the user is sufficiently in band, because out-of-band training contaminates credit assignment and can strengthen state-survival or brittle wrapper-bound habits instead of portable control.

## **2. Zone states**

The Zone Coach classifies the user into one of four states:

- **Ready / In Zone**

- **Flat**

- **Spun Out**

- **Locked In**

These four states route into three practical training classes:

- **Core session**

- **Support session**

- **Recovery / defer**

## **3. Session-counting classes**

### **Core session**

Counts towards the 20.

Criteria:

- Zone state is **Ready**

- full scheduled Capacity session is completed

- normal progression logic is allowed

### **Support session**

Does **not** count towards the 20, but is logged and rewarded separately.

Examples:

- Flat activation + light session

- Spun Out stabilisation micro-session

- Locked In flexibility-reset session

### **Recovery / defer**

Does **not** count towards the 20.

Examples:

- regulation only

- re-check later today

- train tomorrow instead

- park it / minimal safe step only

This fits the G-Loop Coach logic that the system should decide between run now, regulate first, park it, or take a minimal safe step, rather than forcing the same action every time.

## **4. Definitive Zone-routing rules**

### **A. Ready / In Zone**

**Route:** full training day  
**Capacity dose:** 10 blocks  
**Progression:** full  
**Counts towards 20:** yes

Allowed:

- wrapper swap

- fast-speed confirmation

- n-increase if earned

- scheduled tests if due

Coach output:

- **Run now**

- **Full route available**

This is consistent with the standard Capacity session structure of 10 blocks and the normal progression ladder built around last-3-block consistency, speed, and wrapper variation.

### **B. Flat**

**Route:** activate first, then light session  
**Capacity dose:** 4â€“6 blocks  
**Progression:** hold only  
**Counts towards 20:** only if the user re-checks into Ready and then completes a full session; otherwise no

Allowed in the light session:

- normal speed only

- established family only

- no fast-speed confirmation

- no n-increase

- no probe-family session

Preferred families:

- XOR

- AND

Coach output:

- **Regulate first**

- **Then run a lighter session**

- **If still flat, come back later today or tomorrow**

This matches the broader Trident rule that viability comes first and that high drift or weak control should route to simplification rather than intensification.

This follows the signal-board logic that high drift and unstable mismatch should narrow, simplify, and stabilise rather than grind harder.

**Route:** regulate first, then short flexibility-reset session  
**Capacity dose:** 4â€“5 blocks

Preferred family choices:

- XOR non-categorical or conceptual

- basic Interference if appropriate

Coach output:

- **Break rigidity before load**

- **Shift wrapper, lower pressure, restore flexibility**

This is consistent with the Trident distinction between adaptive tightening and maladaptive locking-in, and with the control-mission aim of recovering viable, flexible execution rather than forcing output under brittle conditions.

## **6. Re-check rule**

The app should allow one **short re-check** after regulation for Flat, Spun Out, or Locked In states.

If the re-check changes the user to **Ready**, the app may unlock the full 10-block Capacity session, and that session can count as a **core session**.

If the user remains out of zone, the app should stay in:

- support-session mode, or

- recovery / defer mode

This is important because the goal is not to punish bad days, but to protect the integrity of the encoding phase while still preserving habit continuity. The G-Loop Coach explicitly supports a state-gated adaptation model rather than one rigid route every day.

## **7. Capacity progression constraints by route**

### **Core session**

Use the standard Capacity logic:

- last-3-block consistency governs local routing

- average session n-back governs session-level trend

- wrapper swap is preferred before speed increase

- speed increase is preferred before n-increase

- recovery route triggers if late-session collapse or inconsistency appears

### **Support session**

Use a restricted logic:

- no new progression credit

- no permanent unlock progress

- no probe-family unlock

- variant hold is allowed

- logging still occurs

### **Recovery / defer**

No Capacity progression logic is applied.

## **8. Credits and rewards**

Do **not** treat support sessions as wasted. They should still earn **support credit**, but not **core encoding credit**.

Recommended split on the Hub:

- **Core Rewire progress:** 8 of 20 core sessions

- **Support work:** 3 support sessions

- **In-zone time this week**

- **Zone check-ins passed**

This preserves motivation while staying scientifically honest about what counts as a true encoding repetition. It also fits the scaling logic that some telemetry belongs to the shared kernel while pack-specific progress indicators can vary.

### **Allowed**

### **Postpone**

## **11. Definitive one-paragraph spec**

**The 20-Day Rewire is defined as 20 core encoding sessions, not merely 20 calendar days. A core session counts only when the user is in zone and completes the full scheduled Capacity session. Flat, Spun Out, and Locked In states route first to regulation, then either to a support session or defer. Support sessions are logged and rewarded, but do not count towards the 20-session encoding total. After regulation, the app may allow a brief re-check; if the user re-enters Ready and then completes a full session, that session counts. This preserves the habit while protecting the integrity of the motor-habit encoding phase and keeping progression tied to viable, in-band learning conditions.**
