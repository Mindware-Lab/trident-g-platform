## Best low-complication rule

**Do not rename URLs unless there is a strong reason.**
You can change the **nav labels** and **page content** first.

So:

* keep `/pricing/` but label it **Plans**
* keep `/tools/` for now but label it **Programme**
* keep `/learn/` as **Learn**, and organise it better internally

Later, if you want, you can create:

* `/programme/` as the canonical page
* 301 redirect `/tools/` → `/programme/`

That is the only rename I’d consider worth doing.

# Anti-redundancy rule

To keep the site simple, each page should have **one primary owner job**:

* **Home** = value + route
* **About** = credibility + mission
* **Start** = quickest decision and entry
* **Programme** = how it works
* **Proof** = evidence and boundaries
* **Plans** = commercial routes
* **Coaching** = support layers
* **Learn** = authority content
* **FAQ** = friction reduction

That will stop the same material appearing everywhere.

# Final recommendation

If you want the least complicated implementation:

1. **Leave the URL structure mostly alone**
2. **Repurpose `/tools/` as the Programme page**
3. **Keep `/pricing/` but label it Plans**
4. **Use `/learn/` as the umbrella hub for Research + HAGI + blog + wiki**
5. **Move old standalone-app and launch-pass language out of Home and Plans**
6. **Make Programme + Proof the two main explanatory pages**

That gives you a clean, non-redundant site architecture without forcing a lot of technical churn.
