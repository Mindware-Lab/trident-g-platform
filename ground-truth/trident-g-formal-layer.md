## Appendix X. The Trident-G formal layer: operators, tactics, strategies, mission grammar, and relational reasoning

### X0. Purpose

This appendix defines the formal action layer of Trident-G in its updated form. Its purpose is to clarify how the framework distinguishes the primitive move-language of the system from the broader local and global structures through which those moves are deployed in real cognition and intelligent action.

The central design problem is to preserve a compact operator set while still giving a clear account of how people actually work through structured problems. In Trident-G, this requires a distinction between primitive operators, local tactic patterns, broader strategies for approaching a problem space, mission-specific sequencing rules, and the live relational bridge that makes these structures runnable under real conditions.

The result is a single integrated formal layer within the wider architecture of Trident-G. This layer is not the whole architecture. It sits within the larger system that also includes mission frames, control missions, meta-dials, semantic structure, syntax, subjectivity, allostatic regulation, and runtime capacity. But it is the main mindware layer by which the G-Loop becomes executable in structured, inspectable form.

A further clarification is important. This layer is not only about finding a correct next move. It is also about constructing and navigating a problem space, extracting gist, selecting deeper invariants, pressure-testing what appears promising, and compiling what survives into reusable competence.

### X1. The core hierarchy

The cleanest formal hierarchy in the updated architecture is:

**mission frame → strategy → tactic → operator**

This hierarchy should be interpreted as follows.

**Mission frame** defines what the current run is for. It sets the highest applied purpose of the loop.

**Strategy** defines the broader approach being used within that mission frame to navigate the current problem space.

**Tactic** defines a local reusable composition of operators used to make progress at a particular point in that space.

**Operator** defines the primitive mindware move: the smallest reusable formal action that can be recombined across frames, syntaxes, wrappers, and domains.

This hierarchy allows the architecture to stay clean. The primitive layer remains compact. Richer problem-solving structure can still be described. And the system avoids inflating the Y-operator set with what are better treated as local composites, broader approaches, or governance functions.

### X2. Place in the wider architecture

Within the wider Trident-G stack, this formal layer sits beneath the purpose layer and above the runtime layer.

Above it sit:

meta-missions

meta-dials

mission frames

control missions

feedback signals

Below it sit, or alongside it in the wider architecture:

the semantic layer

the syntax layer

the subjective interface

the allostatic viability layer

the cognitive runtime

the fast relational runtime

This means the formal layer should not be treated as a complete theory of cognition in itself. It is the structured mindware implementation layer of the G-Loop: the part that specifies what formal moves are available, how they are composed locally and globally, how they are sequenced within a frame, and how they become live through relational processing.

### X3. Operator algebra

#### X3.1 Definition

The operator algebra is the primitive move-language of Trident-G. It is the smallest usable set of rule-like mindware moves by which the G-Loop can set rigour, build and transform structure, test and discriminate, commit action, protect viability, and validate portability.

Operators are not the raw processing substrate itself. They are formal moves that act on representational objects such as maps, chunks, schemas, claims, hypotheses, action candidates, plans, cues, and external artefacts. Working memory and attention keep these objects active and manipulable. The operators specify what to do with them.

The operator set should remain compact. A candidate primitive should be included only if it can function as a reusable rule-like move across multiple mission frames, syntaxes, wrappers, and domains without depending on one topic or one problem family.

#### X3.2 Canonical operator set

The canonical operator set remains the Y1–Y16 family.

**A. Rigour and viability gating**

**Y1 — Orient**
Set the rigour budget for the loop.

**Y2 — Recover**
Restore workable corridor occupancy when the loop has become noisy, overheated, flat, or otherwise untrustworthy.

**Y3 — Externalise / Escalate**
Offload cognition into stable scaffolds or escalate when clean in-head continuation is no longer viable or safe.

**B. Representational construction and transformation**

**Y4 — Map**
Build the current relational problem space.

**Y5 — Chunk**
Compress a coherent subgraph into a manageable unit.

**Y6 — Unpack**
Reopen a chunk when it fails, hides dependencies, or proves too coarse.

**Y7 — Analogise**
Transfer structure by role rather than surface similarity.

**Y8 — Abstract**
Extract a higher-order invariant, schema, or organising rule.

**Y9 — Instantiate**
Bind a schema, script, or abstract structure to the present case strongly enough to generate implications, expectations, or next moves.

**Y10 — Re-represent**
Shift framing or level while preserving essential structure.

**Y11 — Meta-map**
Map relations between maps, models, scripts, or frameworks.

**C. Testing and discrimination**

**Y12 — Evaluate**
Assess whether a claim, relation, assumption, or action candidate is good enough to trust.

**Y13 — Discriminate / Probe**
Design or run the cheapest, highest-value test that best separates competing possibilities.

**Y14 — Validate portability**
Test whether a candidate script or structure actually travels across syntaxes, wrappers, stakes, and boundary conditions.

**D. Commitment and meta-control**

**Y15 — Satisfice / Commit**
Stop hovering and choose the next move once the signal is good enough for the current rigour budget.

**Y16 — Level control / anti-regress**
Regulate movement between levels of representation so the loop does not get trapped in over-abstraction, over-detail, or endless regress.

#### X3.3 Design rule

The following should not be added as extra primitive operators:

working-memory holding

attention protection

explore versus exploit

meaning versus moves

banking as such

meta-missions

meta-dials

tactics

strategies

reasoning styles

These belong elsewhere in the architecture. The operator layer should remain the primitive move-language of the kernel.

### X4. Tactic layer

#### X4.1 Definition

Tactics are local, reusable compositions of operators. A tactic is more specific than a broad strategy but more complex than a single operator. It is a structured local move used to make progress at a particular point in a problem space.

Tactics are useful because much of real intelligent action is not best described at the level of a single primitive move. A person often narrows a search space, stabilises a working hypothesis, perturbs the situation to reveal hidden structure, or guards the connectivity of a larger map. These are not new primitives. They are patterned local composites.

Tactics therefore sit above operators and below strategies.

#### X4.2 Canonical tactics

The current canonical tactics include:

**Constraint-induced narrowing**
Use evaluation and probing to eliminate impossibilities and tighten the current space.

**Strategic perturbation**
Use a small, informative test to reveal hidden structure or separate rival explanations.

**Bidirectional constraint squeezing**
Build the space from both ends, hold the goal state explicitly, and shrink the route space until a workable bridge appears.

**Working-hypothesis stabilisation**
Commit defeasibly to the best current hypothesis so that it can be tested, updated, or abandoned cleanly.

**Forced-move commitment**
When the space is sufficiently constrained, commit to the only viable next move rather than hovering.

**Global connectivity guarding**
Preserve the viability of the whole structure by checking whether crucial nodes, paths, or bottlenecks remain intact.

**Confidence-calibrated updating**
Update belief or policy in proportion to evidence quality and reliability rather than by mere emotional force or local fluency.

These tactics are best treated as reusable local patterns of problem-space work executed through the primitive operator language. 

### X5. Strategy layer

#### X5.1 Definition

Strategies are broader problem-space approaches that select and sequence tactics in service of the current mission frame. They are not the highest purpose of the run, and they are not primitive moves. They are broader approaches to how the loop will search, narrow, restructure, or validate within the present problem.

A strategy therefore answers questions such as:

What broad approach is being used here?

Are we mainly narrowing by deduction, exploring rival models, selecting probes, searching routes, updating beliefs, or preserving global connectivity?

How should local tactics be chosen and sequenced to fit the current problem space?

Strategies sit above tactics and below mission frames.

#### X5.2 Canonical strategy families

The current strategy families include:

**Deductive strategies**
Necessary consequence, exclusion, contradiction handling, constraint propagation, counterexample testing.

**Exploratory model-space strategies**
Abduction, counterfactual simulation, analogical reasoning, causal structure search.

**Probe-selection strategies**
Value-of-information, expected information gain, discriminating test design.

**Search / route strategies**
Means–ends reasoning, bidirectional search, operator sequencing.

**Update strategies**
Bayesian or statistical updating, confidence-calibrated updating.

**Structural-map strategies**
Graph-theoretic reasoning, connectivity or reachability reasoning.

These strategies help describe how the loop searches a problem space without changing the primitive architecture of the kernel. 

### X6. Mission grammar

#### X6.1 Definition

Mission grammar is the set of frame-specific well-formedness rules that governs how operators, tactics, and strategies should be sequenced within a given mission frame.

The same operator algebra can be used across frames, but the grammar differs because what counts as a good map, a good tactic, a good test, a good update, a good gist, and a good stop differs between understanding, argumentation, choice, execution, and learning.

Mission grammar therefore supplies the sequencing discipline that turns the formal move-language into a good run for a particular purpose.

#### X6.2 Functional role

Mission grammar answers questions such as:

What sequence of moves is appropriate in this frame?

What counts as good progress here?

When is the loop ready to test, commit, or stop?

What sort of tactic is appropriate at this stage?

What sort of strategic approach fits the present mission?

The operator algebra supplies the vocabulary of moves. Mission grammar supplies the frame-sensitive rules for assembling them into a useful and legitimate run.

### X7. Reasoning calculus

#### X7.1 Definition

Separate from the language of thought and action, though operating within it, is the reasoning calculus.

**Reasoning calculus = inferential transformations run within that formal layer**

The framework distinguishes the formal move from the kind of inference used during that move. The operator says what move is being made. The reasoning style says how inference is being carried out during that move.

#### X7.2 Cross-cutting reasoning styles

Reasoning styles remain cross-cutting overlays rather than extra primitives. The canonical styles include:

deductive reasoning

inductive reasoning

abductive reasoning

analogical reasoning

Bayesian or statistical updating

value-of-information reasoning

These may be treated as tags on operator runs, tactic executions, or strategic approaches.

This yields the cleanest updated formal architecture:

operator algebra = what primitive moves exist

tactic layer = what local reusable composites are available

strategy layer = what broader problem-space approaches are available

mission grammar = what sequences are well-formed for this frame

reasoning calculus = what inferential transformations are being executed within those moves and sequences

### X8. Relational processing and reasoning as the live bridge

Relational processing and reasoning should not be treated as a rival formal layer. It is the live bridge function that makes the whole formal system runnable.

It is the active process that lets the system hold, compare, transform, and test relations rather than merely react to surface cues or retrieve fixed routines. In practical terms, it is what allows the loop to work with structure: what connects to what, what constrains what, what predicts what, what route remains open, what blocks what, and what matters most under current stakes.

Within the wider architecture, relational reasoning:

operates over the semantic layer of ontology, situation map, and affordance surface

is expressed procedurally through the operator algebra

is organised locally through tactics

is guided more broadly through strategies

is sequenced by mission grammar

is shaped by the subjective interface

depends on the cognitive runtime, especially working memory, attention, stability, interference resistance, re-entry, and viable computation under load

For that reason, relational processing is best understood as the working centre of the G-Loop rather than as an extra primitive or a separate grammar.

#### X8.1 The four bridge capacities

The live bridge depends especially on four capacities:

**Relational fluid intelligence**
The ability to detect, restructure, and work with relations when the situation is novel, uncertain, or changing.

**Working memory**
The temporary workspace that keeps rules, relations, constraints, and task-sets active long enough to be manipulated rather than lost to drift.

**Attention management**
The ability to stabilise what matters, suppress misleading pulls, and keep the task-set from collapsing under distraction or salience capture.

**Relational inference and updating capacity**
The ability to derive consequences from current structure, compare rival models, discriminate between alternatives, and update the model under evidence.

These capacities make the formal layer executable under load and make fluid discoveries bankable when they later survive validation.

### X9. Insight, gist extraction, and compression

Insight in Trident-G is not a separate primitive. It is the emergent result of the formal layer running well under viable conditions.

A concise operator pathway for insight and gist extraction is:

**Y4 Map → Y5 Chunk → Y8 Abstract → Y10 Re-represent → Y11 Meta-map**

with later pressure-testing through:

**Y12 Evaluate → Y13 Discriminate / Probe → Y14 Validate portability → Y15 Satisfice / Commit**

This means that gist extraction is procedurally generated within the formal layer rather than being a mysterious extra outside it.

Compression success often feels like:

relief

coherence

clarity

excitement

confidence

a felt click

These feelings may mark improved fit between representation and structure. But they are not self-validating. Pleasurable compression is a candidate signal of emerging gist, not proof of truth or far transfer.

### X10. Relation to semantics, syntax, and portability

The formal layer does not operate over empty symbols. It acts over semantic structure.

A good map is therefore not only formal structure but meaningful structure: entities, states, constraints, goals, values, routes, traps, and affordances inside a bounded world.

The same deeper structure may also be rendered across multiple syntaxes, including verbal, visuospatial, and numerical form. Syntax is therefore a surface coding layer, not the deep structure itself.

This matters because portability is tested by asking whether a structure, tactic, or script survives variation in syntax, wrapper, stakes, and boundary conditions. Formal power alone is not enough. What matters is whether the deeper invariant has actually been abstracted and can be re-instantiated across contexts.

### X11. Relation to the G-Loop kernel

The G-Loop remains the core kernel running across all these layers. It is still the smallest repeatable cycle that keeps cognition viable inside the defended Ψ corridor while converting successful inference into reusable competence.

What the present appendix adds is a clearer statement of what the formal layer contributes:

the operator algebra tells the loop what primitive moves exist

the tactic layer tells the loop what local reusable composites are available

the strategy layer tells the loop what broader approach is being used

mission grammar tells the loop what sequencing is well-formed for this frame

the reasoning calculus tells the loop what kind of inference is being carried out

relational processing tells the loop how structure is being actively held, transformed, and tested in live cognition

Together, these define the formal action layer of Trident-G without blurring it into the semantic, subjective, allostatic, or runtime layers.

### X12. Canonical summary

The formal layer of Trident-G should be understood as an integrated system composed of five tightly related parts: a primitive operator algebra, a tactic layer of local reusable operator compositions, a strategy layer of broader problem-space approaches, a mission grammar that governs frame-specific sequencing, and a reasoning calculus that specifies the inferential transformations run within those structures. This formal layer does not stand alone. It becomes live through relational processing and reasoning, which acts as the bridge between semantic structure, subjective valuation, runtime capacity, and viable action. In this way, Trident-G preserves a compact primitive move-language while still accounting for richer problem-space navigation, gist extraction, portability testing, and the conversion of fluid reasoning into reusable competence.

General rule: **mission frames define purpose, strategies define broader approach, tactics define local reusable composites, operators define primitive moves, reasoning styles define how inference is carried out, and relational processing is the live bridge that makes the whole formal layer runnable.**
