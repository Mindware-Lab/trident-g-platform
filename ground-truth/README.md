# Trident-G Source Documents README

This folder contains the current source documents for the updated Trident-G system. Together, these documents define the architecture, control logic, operator language, relational bridge, far-transfer protocol, and programme-scaling rules for the Trident-G platform.

The documents should be read as parts of one shared system rather than as separate theories. Across the set, the central design rule is consistent: **the kernel remains the same while mission foreground, syntax, wrappers, telemetry, and product framing may vary**.

---

## Included documents

### 1. `Trident G - G Loop.docx`
Defines the **G-Loop** as the core intelligence kernel of Trident-G. The G-Loop is presented as the smallest repeatable cycle that keeps cognition viable within the defended Ψ corridor while converting successful inference into reusable competence. The document clarifies the recurring sequence of corridor check, mission framing, salience-based rigour setting, map building, relational inference, probing and validation, commitment, updating, and conservative banking. It also establishes that the G-Loop is not merely a training loop, but a domain-general control-and-learning kernel that can organise training, decision workflows, research systems, and human-plus-tool environments.

### 2. `Trident G - G Loop Signals.docx`
Defines the compact operational **signal board** used to govern the loop in practice. The core signals are load, drift, mismatch, unexpected mismatch, salience, complexity, and conflict. The document distinguishes core state and error signals from routing, budgeting, and map-management signals, and clarifies their different roles in the architecture. In particular, it establishes mismatch as the main policy-updating signal, unexpected mismatch as the gate for deeper restructuring, salience as the rigour-budget dial, complexity as the map-weight dial, and conflict as a cue source rather than a master scalar.

### 3. `Trident G - High Level Architecture.docx`
Sets out the **high-level layered architecture** of Trident-G. It distinguishes mission frames, the language of thought and action, reasoning calculus, semantic layer, syntax layer, subjective interface, Ψ-allostatic geometry, and cognitive runtime. This document shows how the same G-Loop can be run across different domains, data sources, and action surfaces while preserving a common control-and-inference logic. It also clarifies that subjectivity is architecturally central as the lived interface of valuation, syntax is the surface coding layer rather than the deep structure, and runtime makes the whole system executable under load.

### 4. `Trident G - Scaling for Programmes.docx`
Defines how Trident-G should **scale across products, programmes, and domain packs** without fragmenting its architecture. The central rule is that the kernel, formal layer, runtime substrate, and far-transfer discipline remain fixed, while the foreground mission family, operator emphasis, reasoning emphasis, syntax library, semantic wrappers, telemetry, and user-facing framing may vary. The document provides the clearest basis for building multiple standalone products and later integrated flagships on top of one shared engine.

### 5. `Trident G - Missions Framework and Trident.docx`
Defines the applied **four-layer mission distinction** used in programme design. The four applied layers are mission frames, control missions, meta-dials, and feedback signals. Mission frames define what the loop is being run for. Control missions keep the loop viable and repeatable. Meta-dials shape the slower geometry and ecology of cognition. Feedback signals indicate what kind of update is needed now. The document also ties this applied design back to the deeper Trident metaphor: shaft as banked Gc, centre tine as live Gf, and outer tines as explore and exploit, all governed by the meta-missions of Viability and Expansion.

### 6. `Trident G - Language of Thought.docx`
Defines the **operator algebra** of Trident-G. The canonical primitive set is the Y1-Y16 family, treated as compositional mindware moves rather than raw capacity functions. These operators cover rigour setting, recovery, externalisation, map-building, chunking, unpacking, analogy, abstraction, instantiation, re-representation, meta-mapping, evaluation, probing, portability validation, commitment, and level control. The document also clarifies that deductive, inductive, abductive, analogical, Bayesian, and value-of-information reasoning are not additional primitive operators, but cross-cutting inference styles layered onto operator execution.

### 7. `Trident G - Relational Processing and Reasoning.docx`
Defines **relational processing and reasoning** as the live bridge function at the centre of the architecture. Rather than appearing as an extra top-level layer or a seventeenth operator, relational reasoning is treated as the active process that lets the system hold, compare, transform, and test relations under current constraints. It connects the semantic layer, the runtime, the operator algebra, and eventual banking into Gc. This document is central for understanding how live Gf reasoning becomes validated reusable competence, and why syntax swaps and wrapper swaps matter for true transfer.

### 8. `Trident G - Far Transfer Protocol.docx`
Defines the **far-transfer protocol and reinforcement architecture**. Far transfer is treated not as generic score gain, but as portable operator reliability under changed wrappers, stakes, and real-world conditions. The document specifies a Ψ-gated, probe-first, perturbation-based protocol in which wrapper swaps, syntax swaps, stake swaps, boundary checks, delayed re-checks, and cue-fired deployments act as selection pressures. Reinforcement is hierarchical and mission-first: real mission success outranks local practice reward, mismatch updates policy, unexpected mismatch gates restructuring, and only policies that survive portability testing and real deployment may be banked into Gc.

### 9. `Trident G - Summary.docx`
Provides a compact **canonical synthesis** of the updated Trident-G system. It condenses the shaft-and-tines architecture, the two meta-missions of Viability and Expansion, the applied distinction between mission frames and control missions, and the integrated layered stack from mission frames down to fast relational runtime. This document functions well as a short-form reference for the overall model.

## Integrated system summary

Taken together, the documents define Trident-G as a **domain-general intelligence architecture** organised around one shared G-Loop kernel.

At the deepest level, the architecture is structured as:

- **Shaft = Gc**: banked, validated, reusable competence
- **Centre tine = Gf**: live relational reasoning
- **Outer tines = explore and exploit**: entropy-widening and mutual-information-tightening branches

Above this structure sit two highest-level meta-missions:

- **Viability**: maintain adaptive operation within the defended Ψ corridor so learning, inference, and action remain clean and recoverable
- **Expansion**: enlarge viable dynamic range and portable operator repertoire through exploration, exploitation, and crystallisation into Gc

At the applied level, the architecture distinguishes:

- **Mission frames**: what the loop is being run for
- **Control missions**: what keeps the loop viable
- **Meta-dials**: what slowly shapes cognitive geometry and ecology
- **Feedback signals**: what indicates whether to simplify, probe, recover, update, or restructure

The formal and runtime stack is defined as:

1. Mission frames  
2. Language of thought and action = operator algebra + mission grammar  
3. Reasoning calculus  
4. Semantic layer = ontology + situation map + affordance surface  
5. Syntax layer = verbal + visuospatial + numerical surface forms  
6. Subjective interface = lived salience, value, pressure, and motivation  
7. Ψ-allostatic viability layer  
8. Cognitive runtime = working memory, attention, stability, interference resistance, re-entry, viable bandwidth under load  
9. Fast relational runtime / predictive bridge  

Across the whole set, the same principle is maintained: **the kernel does not change when wrappers, products, training packs, or applied contexts change**.

## Core implementation implications

The documents collectively support the following implementation rules for future Trident-G development:

- Build around **one shared G-Loop kernel**
- Preserve one shared **operator algebra** and **reasoning repertoire**
- Treat **relational reasoning** as the live working bridge, not as a separate extra primitive
- Keep **viability gating** central to all training and deployment
- Use **signals** for routing rather than treating all signals as equivalent dashboard metrics
- Train **capacity** and **mindware** together
- Treat **syntax swaps** as controlled tests of deeper invariant extraction
- Treat **far transfer** as demonstrated portability, not assumed generalisation
- Bank only what survives **swaps, boundaries, cue-fired deployment, and delayed re-checks**
- Scale the system by changing **mission foreground, syntax, wrapper library, coaching emphasis, telemetry, and proof schema**, not by inventing new kernels for each product

**Trident-G is a domain-general intelligence architecture centred on a shared G-Loop kernel that keeps cognition viable within a defended Ψ corridor while converting successful relational inference into portable, validated competence.** The architecture combines mission frames, control missions, meta-dials, fast routing signals, an operator algebra, a reasoning calculus, a semantic layer, a subjective interface, a state-sensitive runtime, and a conservative far-transfer protocol that banks only what survives portability testing.
