# Trident-G summary stack architecture

## 1. Structural architecture

This is the Trident-G form itself.

**Shaft = Gc**  
The banked, reusable support structure, the validated operator library, or crystallised competence.

**Centre tine = Gf**  
The live relational bridge, effortful flexible reasoning in the centre of the loop.

**Outer tines = the two dominant higher-engagement branches**
- explore / entropy-widening
- exploit / mutual-information-tightening

The centre tine is best understood as the live relational bridge in three linked senses: between viability control and banked Gc, between the explore / entropy-widening and exploit / mutual-information-tightening branches, and between live flexible reasoning and the working-memory / attention-control substrate that keeps relational structure active long enough to be transformed, compared, tested, and committed. This is strongly aligned with the Trident-G view that fast SR-like predictive structure functions as a computational middle layer between the fast corridor controller and the portable operator library, while relational reasoning serves as the working centre that moves adaptively between widening and tightening under current constraints.

## 2. Highest-level meta-missions

These sit above the applied mission families and explain what the whole Trident is for.

### Meta-mission 1: Viability

Maintain adaptive allostasis within the defended near-critical Ψ corridor so learning, inference, and action remain clean and recoverable.

### Meta-mission 2: Expansion

Expand viable dynamic range and operator repertoire by using fluid reasoning to coordinate entropy-widening exploration and mutual-information-tightening control, then crystallising what survives into portable Gc.

## 3. Applied mission families

These are not the same thing as the two meta-missions. They are the user-facing, high-language-level purposes for which the Trident is run.

### Mission frames — IQ / Performance
- Understand
- Argue
- Choose
- Plan & Do
- Learn

### Control missions — Resilience / Self-regulation
- Shield attention
- Regulate cadence
- Prioritise missions
- Create distance
- Calibrate mode
- Arbitrate improvisation vs script
- Recover / re-enter
- Externalise / escalate

## 4. Core applied distinctions

For programme design, it is useful to distinguish four applied layers:

- **Mission frames**: what the loop is being run for right now
- **Control missions**: what keeps the loop viable, clean, and repeatable under real conditions
- **Meta-dials**: what slow variables change the geometry, controllability, and ecology of cognition over time
- **Feedback signals**: what fast diagnostic information is telling the loop to simplify, probe, recover, update, or restructure

These layers sit within one shared G-Loop kernel rather than forming separate theories.

**Syntax note:** syntax here means the surface representational form of the same deeper structure. A relation, argument, decision problem, or action sequence can be rendered in verbal, visuospatial, or numerical form without changing its underlying semantic target or operator demand. In training, syntax swaps are a subtype of wrapper swap and are used to test whether performance survives change in surface code rather than depending on one preferred presentation format.

## 6. Stack interpretation

This stack should not be read as a set of isolated modules. The layers are nested and mutually dependent.

- Mission frames specify what success means.
- The language of thought and action specifies what structured move sequences are available.
- The reasoning calculus specifies how inference transforms maps, claims, options, and commitments within those sequences.
- The semantic layer specifies what the system is modelling.
- The syntax layer specifies how the same deeper structure is rendered at the surface.
- The subjective interface specifies how significance, cost, possibility, and threat are lived by the organism.
- The allostatic layer specifies whether the system is in a viable regime to reason cleanly.
- The cognitive runtime specifies what can actually be held, transformed, and executed.
- The fast relational runtime provides the live predictive bridge through which structure can be carried forward and acted upon.

## 7. Kernel note

The G-Loop remains the core intelligence kernel running across all these layers. It is the smallest repeatable cycle that keeps cognition viable inside the defended Ψ corridor while converting successful inference into reusable competence. The kernel remains the same even when mission frames, syntaxes, wrappers, domains, and applied programmes change.

## 8. One-paragraph summary

Trident-G is best understood as a domain-general intelligence architecture organised around a shaft-and-tines structure and run by a common G-Loop kernel. The shaft is banked Gc competence, the centre tine is live Gf relational reasoning as the bridge between viability control, banked Gc, and the explore–exploit branches, and the outer tines are entropy-widening exploration and mutual-information-tightening control. Above the architecture sit the two meta-missions of Viability and Expansion. At the applied level, Trident-G is run through mission frames for IQ / performance and control missions for resilience / self-regulation. Formally, the system is organised as a language of thought and action, composed of operator algebra and mission grammar, together with a reasoning calculus, a semantic layer, a syntax layer, a subjective interface, a Ψ-allostatic viability layer, a cognitive runtime, and a fast SR-like relational runtime. Together these layers explain how Trident-G integrates purpose, modelling, subjectivity, inference, state control, and runtime execution into a single architecture.
